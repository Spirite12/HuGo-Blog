前言：Addressables 与 AssetBundle

Addressables 系统构建于 Unity AssetBundles 之上。如果在项目中使用AssetBundles且不想编写自己的详细管理代码，则这时候可以使用 Addressables来达到效果；

Addressables的相关优点：

  1. 灵活性：可寻址使您可以灵活地调整资产的托管位置。您可以随应用程序安装资产或按需下载它们。您可以在项目的任何阶段更改访问特定资产的位置，而无需重写任何游戏代码。
  2. 依赖管理：系统会自动加载您加载的任何资源的所有依赖关系，以便在系统将内容返回给您之前加载所有网格、着色器、动画和其他资源。
  3. 内存管理：系统卸载和加载资产，自动计算引用并提供强大的分析器来帮助您发现潜在的内存问题。
  4. 内容打包：由于系统映射并理解复杂的依赖链，因此即使您移动或重命名资产，它也能有效地打包 AssetBundles。您可以为本地和远程部署准备资产，以支持可下载内容并减少应用程序大小。

安装

  • 安装路径：Window → Package Manager → Addressables

  • 初始数据：Window → Asset Management → Addressables → Groups

    • 如果没有 Addressables 组，则可以直接点击 Create Addressables Settings 进行初始化

      • 系统会在 Assets 文件夹下创建 AddressableAssetsData 文件夹以及相关数据

    • 如果有 Addressables 组，则可以把该组直接拖进当前窗口进行初始化；

标记资源为 Addressables

前言：Resources 文件夹以及此子目录无法标识 Addressables。如果对 Resources 文件夹进行标识，Unity 则会弹窗提示并使其移除资源文件夹；

  1. Inspector 单个资源标识：
    • 点击任一资源，在 Inspector 面板最上方出现 Addressable 的勾选框，勾选即可标记；
    • 标记成功后：
      • 当前的 Groups 中即可增加当前资源的相关信息；
      • 勾选框后方出现 String 类型的 Addressable 的唯一标识（也是加载地址）；
        1. 标识也可不唯一，Api提供了许多函数来引用这些资源，当然唯一性会让项目更容易保持组织性;
          • 不唯一的例子：
            • 如果您有 Asset 的变体，您可以为所有变体分配相同的地址，并使用标签来区分不同的变体：
            • Asset1：地址：“plate_armor_rusty”，标签：“hd”
            • Asset2：地址：“plate_armor_rusty”，标签：“sd”
            • 如果此时用：LoadAssetAsync 则置灰加载找到的第一个实例
        2. 在脚本中可直接使用 String 标识来加载资源，而不必写路径代码，这样当路径发生变化时，无需重写代码；
  2. Inspector 文件夹资源标识：
    • 点击任一文件夹，在 Inspector 面板最上方出现 Addressable 的勾选框，勾选即可标记；
  3. Addressables Group 标识：
    • 手动将资源或者文件夹拖进 Groups 窗口的 Default Local Group 栏目下即可标识
  4. 代码类型为 AssetReference 类型的引用标识：
    2. 代码写 AssetReference 类型变量，并赋值后 Addressables 资源后，即可打上标识；

    public class Test : MonoBehaviour {
    // 加载资源地址
    public string strAddress;
    // 加载回调
    private AsyncOperationHandle<GameObject> handle;

    void Start() {
        if (strAddress.Length <= 0) {
            Debug.LogError("StrAddress is Null");
            return;
        }
        handle = Addressables.LoadAssetAsync<GameObject>(strAddress);
        handle.Completed += HandleCompleted;
    }
    void OnDestroy() {
        Addressables.Release(handle);
    }
    private void HandleCompleted(AsyncOperationHandle<GameObject> obj) {
        if (obj.Status != AsyncOperationStatus.Succeeded) {
            Debug.LogError($"Asset for {strAddress} failed to load.");
            return;
        }
        Instantiate(obj.Result, transform);
    }
}

    public class Test : MonoBehaviour {
    // 加载的资源引用
    public AssetReference assetReference;
    void Start() {
        if (assetReference.AssetGUID.Length <= 0) {
            Debug.LogError("AssetReference is Null");
            return;
        }
        AsyncOperationHandle handle = assetReference.LoadAssetAsync<GameObject>();
        handle.Completed += HandleCompleted;
    }
    void OnDestroy() {
        assetReference?.ReleaseAsset();
    }
    private void HandleCompleted(AsyncOperationHandle obj) {
        if (obj.Status != AsyncOperationStatus.Succeeded) {
            Debug.LogError($"AssetReference {assetReference.RuntimeKey} failed to load.");
            return;
        }
        Instantiate(assetReference.Asset, transform);
    }
}

      public class Test : MonoBehaviour {

    public List<string> keys = new List<string>() { "Test" };
    // 加载回调
    private AsyncOperationHandle<IList<GameObject>> handle;

            keys,
            addressable => {
                if (addressable != null) {
                    Instantiate<GameObject>(addressable, transform);
                }
            },
            Addressables.MergeMode.Union,
            false
        );
        handle.Completed += HandleCompleted;
    }
    void OnDestroy() {
        Addressables.Release(handle);
    }
    private void HandleCompleted(AsyncOperationHandle<IList<GameObject>> obj) {
        if (obj.Status != AsyncOperationStatus.Succeeded) {
            Debug.LogError($"Asset key failed to load.");
        }
    }
}

加载和释放Asset

  • 加载：
    1. 调用函数：LoadAsset（加载单个资源）、LoadAssetAsync（异步加载单个资源）、LoadAssets（加载多个资源）、LoadAssetsAsync（异步加载多个资源）
    2. 调用说明：调用加载相关的函数，Addressables 会自动加载任何依赖资源。
  • 释放：
    1. 调用函数：Addressables.Release;
    2. 调用说明：
      1. 当不再需要在运行时访问可寻址资源时，您必须需要释放它，否则会写在不干净；
      2. 当系统卸载资产时，Addressables 会卸载其依赖项。除非资产仍在被其他资产使用。
  • 引用计数：
    1. 当加载一个Addressable时，系统会增加引用计数；当资产被释放时，系统会减少引用计数；
    2. 当Addressable的引用计数返回到零时，它就可以被卸载了；
    3. 为了避免“内存泄漏”，当您显式加载可寻址资源时，您还必须在使用完该资源后释放该资源；

Addressables Event Viewer

  • 介绍：工具监视 Addressables 资源的内存管理。该窗口显示应用程序何时加载和卸载资源，并显示所有 Addressables 系统操作的引用计数；

  • 路径：Window → Asset Management → Addressables → Event Viewer

  • 开启：在使用工具前勾选：AddressableAssetSettings → Diagnostics → Send Profiler Events

  • 条件：

    • 在 Groups 窗口内，不同的 Play Mode Script 选择，会对工具监视会有不同的生效；
      • Use Asset Database：对引用计数的监控不太准确；
      • Simulate Groups：无法使用，不考虑资产之间的任何共享依赖关系；
      • Use Existing Build：可在播放模式下的事件查看器中获取最准确的信息

  • 示例：

    1. 使用 Event Viewer 查看界面中对资源的销毁、释放、加载的Addressables的分析情况，具体步骤：

      1. 初始UI界面上，仅有三个按钮，分别点击效果是：

        1. 通过 Addressables 的标签加载预制体；
        2. 使用 GameObject：Destroy：直接删除一个预制体
        3. 使用 Addressables.Release : 释放全部资源

      3. 此次通过标签 “Test” 加载2个预制体，这2个预制体属于同一个Group 下；

      using System.Collections.Generic;
using UnityEngine;
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;
using UnityEngine.UI;

public class Test : MonoBehaviour {
    public Button btnRelease;
    public Button btnDestroy;
    public Button btnInstance;

    // 加载资源的Key列表
    public List<string> keys = new List<string>() { "Test" };
    public List<GameObject> goNewList = new List<GameObject>();

    // 加载回调
    private AsyncOperationHandle<IList<GameObject>> handle;

    void Start() {
        AddBtnListeners();
    }

    void OnDestroy() {
        foreach (var go in goNewList) {
            GameObject.Destroy(go);
        }
        goNewList.Clear();
        if (handle.IsValid()) {
            Addressables.Release(handle);
        }
    }

    private void AddBtnListeners() {
        // 销毁资源
        btnDestroy?.onClick.AddListener(() => {
            int destroyIndex = 0;
            GameObject.Destroy(goNewList[destroyIndex]);
            goNewList.RemoveAt(destroyIndex);
        });
        // 释放资源
        btnRelease?.onClick.AddListener(() => {
            handle.Completed -= HandleCompleted;
            OnDestroy();
        });
        // 加载资源
        btnInstance?.onClick.AddListener(() => {
            if (keys.Count <= 0) {
                Debug.LogError("StrAddress is Null");
                return;
            }
            handle = Addressables.LoadAssetsAsync<GameObject>(
                keys,
                addressable => {
                    if (addressable != null) {
                        var goNew = Instantiate<GameObject>(addressable, transform);
                        goNewList.Add(goNew);
                    }
                },
                Addressables.MergeMode.Union,
                false
            );
            handle.Completed += HandleCompleted;
        });
    }

    private void HandleCompleted(AsyncOperationHandle<IList<GameObject>> obj) {
        if (obj.Status != AsyncOperationStatus.Succeeded) {
            Debug.LogError($"Asset key failed to load.");
        }
    }
}

小内容记录

  1. Asset address：标识可寻址资产的字符串 ID。您可以使用地址作为加载资产的密钥；
  2. addressables_content_state.bin：用于进行内容更新构建，该文件用来识别您更改了哪些资产。您必须为每个已发布的版本保留此文件的副本。如果没有该文件，您只能创建完整的内容版本，而不能创建更新（理解是：只能打整包，不能热更）
  3. Check for Content Update Restictions 内容更新构建
    1. 说明：
      • 内容构建会生成内容目录、目录哈希和包含资产的 AssetBundles
      • 由于资产格式是特定于平台的，因此切换到每个平台前需要进行内容构建
    2. 操作：
      • Addressables Groups → Tools → Check for Content Update Restictions；
      • 会出现一个Win窗口，在这个创建下找到相对应平台文件夹下的 addressables_content_state.bin 文件；
      • 此时会出现更新预览窗口，会列出更新的变更记录，点击 Apply Changes 即可完成内容更新构建；
  4. Play Mode Script：
    1. Use Asset Database：直接从资产数据库加载资产。如果您同时进行代码和资产更改，此选项通常会提供最快的迭代速度
    2. Simulate Groups：如果您正在组织和优化可寻址组本身，此选项会很有帮助。它提供了 Addressables 事件，无需在每次更改后重建完整的内容。
    3. Use Existing Build：从上次的内容构建中加载内容，如果您不更改/更新资产，则可以提供快速迭代周转；
  5. Addressables 组的 Asser Bundle 里面的压缩方式对比
    1. 路径：Projecr → AddressableAssetsData → 相对应Group资产 → Content Packing & Loading → Asset Bundle Compress
    2. 无压缩：加载速度很快、磁盘空间占用最大；
    3. LZ4压缩：加载速度适中、提供最中的包，通常用于本地内容。这是一种基于块的压缩，可以加载部分文件，而无需加载整个文件；
    4. LZMA压缩：加载速度很慢，提供最小的包，通常用于远程内容，但不适用于本地内容；

问题记录

  1. 在安装 Addressables 后运行游戏的时候，会出现如下截图的报错：
    • 原因是：
      1. 在 Addressables Groups → Play Mode Script → 使用的是 Use Existing Build模式
      2. 处在这个模式下，需要先执行 Build 打出 Addressables 资源包，这样代码才能正确加载到资源
      3. Build方式：在 Addressables Groups → Build → New Build → Default Build Script
      4. 打完后便可运行游戏了；
  2. 在更改资源后运行游戏的时候，发现游戏中的资源并没有发生改变：
    • 原因是：
      1. 在 Addressables Groups → Play Mode Script → 使用的是 Use Existing Build模式
      2. 处在这个模式，需要重新 Build 资源；（方式在问题1）
